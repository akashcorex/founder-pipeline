# Crosspost

Distribute content across platforms without posting the same thing twice.

## When to Activate

- A post written for LinkedIn needs an X version (or vice versa)
- A launch, project, or insight needs platform-specific versions
- The user says "adapt this for X" or "post this everywhere"

## Core Rules

1. Do not publish identical copy across platforms.
2. Preserve the builder voice — sharp, honest, first person.
3. Adapt for constraints, not stereotypes.
4. One post = one claim.
5. Do not invent a CTA, question, or moral if the source didn't earn one.

## Workflow

### Step 1: Start with the Primary Version

Use the strongest native version as source:
- A LinkedIn post with full context
- An X thread with a sharp claim
- A build log, project writeup, or launch note

### Step 2: Adapt by Platform

#### X (from LinkedIn)
- Strip context that X-native readers don't need
- Pull the sharpest claim to the front
- Remove LinkedIn formatting habits (paragraph breaks, praise stacking, closing questions)
- Thread only if the argument genuinely needs multiple posts
- No hashtags

#### LinkedIn (from X)
- Add only the context needed for people outside the immediate niche
- Preserve the compression — don't pad to fill space
- Do not turn it into a fake lesson post
- Do not add a closing question just because it's LinkedIn
- Keep the builder voice — don't polish it into corporate

## Banned Patterns

- "Excited to share"
- "Here's what I learned" (just say it)
- "What do you think?" (unless the original earned it)
- Generic "professional takeaway" paragraphs not in the source
- Forced casualness on LinkedIn
- Fake engagement padding on either platform

## Output Format

Return:
- The primary platform version
- Adapted variant for the other platform
- A short note on what changed and why (one line)

## Quality Gate

- Each version reads like the same person under different constraints
- No platform version feels padded or sanitized
- No copy is duplicated verbatim across platforms
- Any extra context added for LinkedIn is actually necessary
